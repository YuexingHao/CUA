The Math, Explained

Six equations that matter

Each formula solves a specific real problem. Below we break down what they mean, why they matter, and how they connect to our codebase.

🏗
1. Hierarchical MDP -- The Agent's World Model
Why it matters: Standard MDPs treat every mouse click as an independent decision. But real tasks have structure -- "search for a product" is a coherent sub-goal, not a sequence of unrelated clicks. The hierarchical MDP gives the agent two levels of decision-making: pick a high-level skill, then execute it with low-level primitives.
M = (S, Alow, Ahigh, P, R, γ)Hierarchical Markov Decision Process
S -- State space
The current screenshot, DOM tree, and task description. Everything the agent can observe.
Alow -- Low-level actions
Raw primitives: mouse clicks at (x,y), keystrokes, scrolls. These are continuous in [0,1]4.
Ahigh -- High-level actions (skills)
Learned abstractions like "search_product" or "filter_and_sort". Each skill is a policy over low-level actions.
P -- Transition function
How the environment changes after an action. In web UIs, this is non-deterministic (pages load, popups appear).
R -- Reward
Task success signal: did the agent complete the objective? Sparse and delayed.
γ -- Discount factor
How much to weight future rewards vs. immediate ones. Typically 0.99 for long web tasks.
Analogy: Restaurant kitchen

A chef (agent) works at two levels. High-level: decide what dish to prepare next ("make risotto"). Low-level: individual cooking steps (chop onions, heat pan, stir). The chef does not think about every knife motion when deciding the menu -- skills let the agent do the same.

high(z | s) search_product filter_and_sort select_purchase LOW-LEVEL POLICY: πlow(a | s, z) click() type() scroll() click() select()
See interaskill/mdp.py -- defines the hierarchical state-action spaces and transition logic.
🎯
2. Normalized Actions -- Resolution Invariance
Why it matters: SKILL.md stores actions as pixel coordinates like "click at (847, 312)." This breaks when the screen resolution changes, the window resizes, or the UI layout shifts. Normalizing all coordinates to [0,1] makes the same policy work on 1080p, 4K, or any resolution -- the agent thinks in relative positions, not absolute pixels.
aprim = (x, y, atype, d) ∈ [0,1]4Normalized continuous action space
x -- Horizontal position
Fraction of screen width. 0.0 = left edge, 1.0 = right edge. E.g., 0.45 = slightly left of center.
y -- Vertical position
Fraction of screen height. 0.0 = top, 1.0 = bottom. E.g., 0.08 = near the top bar.
atype -- Action type
Encoded as a continuous value: 0.0 = click, 0.33 = type, 0.67 = scroll, 1.0 = special.
d -- Action data
Additional parameter: scroll distance, key code, or duration. Normalized to [0,1].
Analogy: GPS vs. street addresses

Street addresses ("123 Main St") only work in one city. GPS coordinates (42.36, -71.06) work everywhere. Normalizing actions is like switching from street addresses to GPS -- the same instruction works regardless of the "city" (screen resolution).

See interaskill/actions.py -- the normalize_action() function converts pixel coords to [0,1] fractions.
🧲
3. InfoNCE Loss -- How Skills Are Discovered
Why it matters: We need to discover skills without any human labels. The InfoNCE loss does this by learning to match trajectories to their correct skill embedding while pushing apart trajectories that belong to different skills. It is the same contrastive approach used by CLIP and SimCLR, adapted here for action sequences instead of images.
Lskill = −Eτ[log exp(f(τ, zτ) / T) / ∑z' exp(f(τ, z') / T)]InfoNCE contrastive loss
τ -- Trajectory segment
A sequence of actions that form a candidate skill (e.g., 3 clicks that perform a search).
zτ -- Positive skill embedding
The "correct" skill label for this trajectory, assigned by clustering.
z' -- Negative skill embeddings
All other skill labels in the batch. The loss pushes τ away from these.
f(τ, z) -- Compatibility score
Dot product between the trajectory encoder output and the skill embedding vector.
T -- Temperature
Controls the sharpness of the distribution. Lower T = harder contrastive signal. We use T=0.07.
Analogy: Sorting mail into mailboxes

Imagine sorting 5,000 letters into 12 mailboxes, but nobody labeled the mailboxes. You compare each letter to every mailbox and put it in the one it matches best. Over time, each mailbox specializes: one gets all the bills, another gets personal letters. InfoNCE does this automatically for action sequences.

See interaskill/discover.py -- the InfoNCELoss class and train_skill_encoder() training loop.
🌐
4. CLIP Transfer -- Cross-Domain Matching
Why it matters: A "Submit" button on Shopify and a "Submit" button on GitHub look different but serve the same purpose. CLIP embeddings capture this semantic similarity, allowing the agent to recognize that a skill learned on one website can be reused on another -- even if the visual appearance is completely different.
Sim(sA, sB) = cos(φCLIP(sA), φCLIP(sB)) > θSemantic state matching via CLIP
sA, sB -- Screenshots from two different domains
E.g., a product listing on Amazon vs. a product listing on eBay.
φCLIP -- CLIP image encoder
Pre-trained vision-language model that maps images to a semantic vector space. We use ViT-B/32.
cos(·, ·) -- Cosine similarity
Measures angle between two vectors. Range [-1, 1]. Higher = more similar.
θ -- Transfer threshold
Minimum similarity required to reuse a skill across domains. We use θ = 0.75.
Analogy: Universal language translator

Imagine you speak only English and visit Japan. You cannot read the signs, but a universal translator tells you "this sign means Exit, just like back home." CLIP is that translator for UI elements -- it recognizes the meaning behind different visual appearances.

See interaskill/transfer.py -- the clip_similarity() function and should_transfer_skill() decision logic.
🔗
5. Option-Value Recursion -- Composing Skills
Why it matters: The agent needs to decide not just which skill to use, but when to stop one skill and start another. The option-value recursion (from the Options framework by Sutton, Precup, Singh 1999) gives the agent a principled way to evaluate "should I keep doing this skill, or switch to something better?"
Q(s,z) = r(s,z) + γEs'[(1−βz)Q(s',z) + βz maxz' Q(s',z')]Option-value recursion
Q(s, z) -- Option value
Expected future reward of executing skill z in state s until it terminates.
r(s, z) -- Immediate reward
Reward obtained from executing one step of skill z in state s.
βz(s') -- Termination probability
Probability that skill z should stop in state s'. Learned from data: e.g., "stop searching once results appear."
(1 - βz) -- Continue probability
If the skill does not terminate, continue executing it and get Q(s', z).
maxz' Q(s', z') -- Best next skill
If the skill terminates, pick the best next skill to execute.
Analogy: Relay race

Each runner (skill) runs their leg until they reach the handoff zone (termination condition). They do not stop mid-stride; they complete their segment. The baton pass (βz) determines when one runner hands off to the next. The coach (Q function) decides who runs each leg for the fastest total time.

See interaskill/policy.py -- the OptionValueFunction class implements Q(s,z) with learned termination β.
📐
6. Jacobian Orthogonality -- Keeping Skills Distinct
Why it matters: Without regularization, learned skills tend to collapse -- multiple skill dimensions end up encoding the same behavior. The Jacobian orthogonality penalty forces each skill dimension to control a different aspect of behavior, ensuring the full skill space is used and skills remain distinguishable from each other.
Lortho = ‖JzTJz − I‖2FJacobian orthogonality regularization
Jz -- Jacobian matrix
Matrix of partial derivatives: how much each action output changes when each skill dimension changes. Size: (action_dim x skill_dim).
JzTJz -- Gram matrix
Measures the correlation between skill dimensions. If two skill dims affect actions the same way, their entry will be large.
I -- Identity matrix
The target: each skill dimension is perfectly independent (correlation = 0 with all others, self-correlation = 1).
‖·‖2F -- Frobenius norm squared
Sum of squared differences between JTJ and I. Zero means perfect orthogonality.
Analogy: TV remote control

Imagine a remote where volume and channel buttons both change the volume. That is mode collapse -- two controls doing the same thing. The orthogonality loss is like a rule that says "each button must do something different." Volume changes volume, channel changes channel, brightness changes brightness -- each skill dimension controls one independent aspect of behavior.

See interaskill/regularizers.py -- the jacobian_orthogonality_loss() function, called during skill encoder training.
Full Pipeline

From raw trajectories to executable skills

1
Trajectory Segmentation

Compute Δat = ||at - at-1||. Detect boundaries where Δat > θ. Each segment = candidate skill.

2
Segment Representation

Model each segment as Gaussian mixture p(a|segment) = ∑πmN(a; μm, Σm). Compare via Wasserstein distance.

3
Hierarchical Clustering

Agglomerative clustering on Wasserstein distance matrix. Automatically discovers k skills without specifying k.

4
Contrastive Embedding

InfoNCE loss maps clusters into continuous latent space z ∈ R16. Multimodal encoder ingests vision + text + action.

5
Hierarchical Policy

Macro policy πmacro(z|s) selects skills. Each skill executes until termination βz(s) fires. Options framework.
